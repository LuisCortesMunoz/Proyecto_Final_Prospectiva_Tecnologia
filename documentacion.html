01 · Resumen ejecutivo

¿Qué es LadderVoice?

LadderVoice, también referido como LAVO, es un sistema web pensado para facilitar la programación y operación de lógica Ladder para PLC mediante lenguaje natural y voz. El usuario dicta o escribe una instrucción, el sistema la interpreta con IA, genera una configuración lógica validable y la transforma en una representación Ladder visual.

El proyecto integra un frontend desplegable en GitHub Pages, un backend con FastAPI, servicios de IA para transcripción y generación lógica, y un puente local capaz de enviar configuraciones al PLC Horner XL4 / XC1E5 mediante Modbus TCP.

02 · Objetivo del proyecto

Programar lógica de control usando lenguaje natural

El objetivo principal fue construir una herramienta que permitiera pasar de una instrucción humana, como “con el botón I1 enciende la lámpara verde y con I2 apágala”, a una lógica de control estructurada, visible y eventualmente aplicable a un PLC físico.

Entrada por voz

El usuario puede grabar una instrucción desde el navegador y convertirla a texto.

Interpretación con IA

El texto se manda a un modelo LLM para obtener una configuración lógica estructurada.

Visualización Ladder

El frontend compila esa lógica a un programa Ladder visual, validado y entendible.

Una decisión importante fue evitar que la IA dibujara directamente la geometría Ladder. En su lugar, la IA solo interpreta la intención y devuelve un JSON lógico simple; después, el frontend valida y compila ese JSON de manera determinista.
03 · Arquitectura general

Separación entre interfaz, IA y control físico

La arquitectura se dividió en capas para evitar que una sola parte del sistema tuviera demasiada responsabilidad. El navegador se encarga de la interfaz, el backend de IA se encarga de transcribir e interpretar, y el puente local se encarga de comunicarse con el PLC.

Capa Responsabilidad Archivos principales
Frontend web Interfaz visual, asistente, editor Ladder, validación, render y simulación. index.html, asistente.html, ladder.html, assets/js/ladder/*
Backend IA Transcripción de audio, generación de JSON lógico, historial y memoria por feedback. app.py, context_json/contexto.json, memoria/ejemplos.json
Compilador Ladder Transforma el JSON lógico simple en un esquema Ladder renderizable. logicToSchema.js, validate.js, renderer.js
Puente PLC Escribe registros en el PLC por Modbus TCP sin reprogramar todo el PLC desde Python. plc_maestro.py, endpoint /aplicar-plc

04 · Flujo de operación

De una frase hablada a una lógica Ladder

1
Captura de instrucción El usuario escribe o dicta una instrucción desde la página. En el flujo de voz, el navegador graba audio usando la interfaz web.
2
Transcripción STT El audio se envía al backend mediante /transcribir, donde se utiliza Groq con Whisper para convertir el audio a texto en español.
3
Interpretación lógica El texto se manda a /generar-logica. El modelo LLM interpreta la intención y devuelve un JSON lógico simple, no una geometría Ladder.
4
Validación El frontend valida que las entradas, salidas, timers, contadores y modos existan dentro del perfil del dispositivo.
5
Compilación visual El compilador del frontend transforma la configuración lógica en rungs, contactos, ramas, bobinas y elementos visuales.
6
Aplicación al PLC Cuando se requiere control físico, el backend local envía la configuración al PLC escribiendo registros mediante Modbus TCP.
05 · Evolución y cambios realizados

Cómo fue madurando el sistema

El desarrollo no nació como una arquitectura definitiva. Se fue ajustando conforme aparecieron problemas reales de integración, validación, generación de código y conexión con el PLC.

Etapa Cambio realizado Motivo
Definición inicial Se planteó un agente capaz de recibir voz y traducirla a acciones de automatización. Reducir la fricción de programar PLCs manualmente.
Selección del PLC Se definió el uso del PLC Horner XL4 / XC1E5 con Cscape. Trabajar con el hardware real disponible en laboratorio.
Backend de IA Se agregó FastAPI, endpoints de transcripción, generación Ladder y generación lógica. Separar la interfaz web de los modelos de IA y de las claves API.
Groq STT Se integró Whisper desde Groq para transcribir audio en español. Obtener texto confiable a partir de comandos hablados.
Generación Ladder inicial Se probaron respuestas tipo JSON Ladder completo con rungs, filas y elementos. Permitió validar el concepto, pero era demasiado libre y podía generar estructuras inconsistentes.
Arquitectura única Se cambió a un contrato de JSON lógico simple. La IA solo interpreta; el frontend compila de forma determinista y evita renderizar lógica falsa.
Editor visual Se consolidó el editor Ladder con render, simulador, validación y generación por chat. Hacer que el usuario vea claramente la lógica antes de aplicarla.
Puente PLC Se implementó plc_maestro.py para escribir registros %R por Modbus TCP. No reprogramar el PLC completo desde Python, sino cambiar una configuración maestra.
Memoria y feedback Se agregó almacenamiento de ejemplos aceptados, corregidos o rechazados. Mejorar respuestas futuras con ejemplos validados por el usuario.
Copiloto de chat Se incorporó un copiloto de chat con perfiles de esfuerzo: genérico, instantánea, media y alta. Ofrecer un asistente para explicación, depuración y apoyo técnico.
06 · Backend y APIs

Servicios usados para IA, voz y control

El backend principal está construido con FastAPI. Su función es recibir peticiones desde el frontend, comunicarse con los modelos de IA, guardar contexto y, cuando se ejecuta localmente, servir como puente hacia el PLC.

Endpoints principales

Endpoint Función
POST /transcribir Recibe audio y devuelve texto usando Groq Whisper.
POST /voz-a-ladder Flujo completo de audio a Ladder usado durante las primeras pruebas.
POST /generar-ladder Genera una estructura Ladder más directa a partir de texto.
POST /generar-logica Endpoint principal actual: texto a JSON lógico simple.
POST /aplicar-plc Aplica la configuración al PLC físico mediante Modbus TCP.
GET /plc/escanear Busca dispositivos Modbus TCP en la red local.
POST /feedback Registra si una respuesta fue aceptada, corregida o rechazada.
GET /memoria Muestra la memoria de ejemplos usados para mejorar futuras generaciones.

APIs utilizadas

  • Groq API: usada para transcripción STT con whisper-large-v3 y para generación lógica con un modelo LLM configurado desde variables de entorno.
  • Groq para copiloto de chat: usado para responder dudas técnicas mediante perfiles de esfuerzo.
  • Ollama: considerado y probado como alternativa local para asistencia técnica y modelos locales.
  • Modbus TCP: usado para comunicar Python con el PLC Horner mediante registros.
  • APIs del navegador: usadas para grabación de audio, interacción web y posible síntesis de voz.
Variables relevantes del backend:

GROQ_API_KEY       → llave para generación con LLM
GROQ_API_KEY_stt   → llave para transcripción STT
GROQ_MODEL         → modelo de generación lógica
GROQ_STT_MODEL     → modelo de transcripción, por ejemplo whisper-large-v3
ADMIN_TOKEN        → token para rutas administrativas
CONTEXTO_JSON      → ruta del contexto de referencia
MEMORIA_JSON       → ruta de memoria de ejemplos
07 · Frontend

Interfaz web, asistente y editor Ladder

El frontend está desarrollado principalmente con HTML, CSS y JavaScript modular. Se decidió mantener una arquitectura ligera para que pudiera desplegarse como sitio estático en GitHub Pages.

Páginas principales

Página Uso
index.html Página del copiloto IA y entrada principal del proyecto.
asistente.html Interfaz enfocada en el asistente de voz.
ladder.html Editor visual Ladder con panel de chat, simulación y herramientas.
about.html Explicación general del proyecto.
documentacion.html Esta nueva página: documentación técnica, evolución y créditos.

Módulos relevantes

  • assets/js/ladder/generate.js: punto único para generar programas Ladder desde texto o JSON lógico pegado.
  • assets/js/ladder/gen-bridge.js: expone window.LadderGen para que páginas no modulares puedan usar el mismo flujo.
  • assets/js/ladder/validate.js: valida el JSON lógico antes de renderizar.
  • assets/js/ladder/compiler/logicToSchema.js: compila la lógica a un esquema Ladder visual.
  • assets/js/ladder/renderer.js: dibuja los rungs en pantalla.
  • assets/js/ladder/simulator.js: permite probar la lógica desde el navegador.
08 · PLC, Cscape y registros maestros

Control del PLC sin reprogramarlo desde cero cada vez

Una de las decisiones más importantes fue dejar en el PLC un programa Ladder base o maestro. En lugar de generar y descargar un programa nuevo en cada instrucción, Python actualiza registros de configuración. El PLC lee esos registros y cambia su comportamiento lógico.

PLC objetivo

  • PLC: Horner XL4 / XC1E5.
  • Software: Cscape.
  • Comunicación: Modbus TCP, puerto 502.
  • Entradas principales: I1, I2, I3, I4 e I7.
  • Salidas principales: Q10, Q11 y Q12.

Idea de registros maestros

El archivo plc_maestro.py traduce el JSON lógico a escrituras en registros. Cada salida tiene un bloque de registros propio, por ejemplo Q10, Q11 y Q12. Ahí se guarda el modo de operación, entradas asociadas, paro, habilitación, timer y contador. 

Ejemplo conceptual:

Q10 → bloque de registros %R10..%R19
Q11 → bloque de registros %R20..%R29
Q12 → bloque de registros %R30..%R39

Modos:
0 → apagado
1 → directo
2 → enclavado

Capas opcionales:
timer   → on_delay o pulse
counter → up o up_held
reset   → entrada física configurable
Esta estrategia hizo el proyecto más viable: Python no sustituye a Cscape ni descarga continuamente programas completos. Python solo modifica parámetros y el PLC ejecuta la lógica maestro ya cargada.
09 · Uso de inteligencia artificial

IA como apoyo para voz, generación y desarrollo

La IA se usó en dos niveles: como parte funcional del sistema y como apoyo durante el desarrollo. Dentro del sistema, ayuda a transcribir voz y traducir lenguaje natural a configuraciones lógicas. Durante el desarrollo, diferentes asistentes ayudaron a estructurar, depurar y mejorar el proyecto.

IA dentro del sistema

  • STT: Groq Whisper convierte audio a texto.
  • LLM: Groq interpreta instrucciones y genera JSON lógico simple.
  • Copiloto de chat: Groq responde dudas y apoya con explicación técnica mediante perfiles de esfuerzo.
  • Ollama: explorado como alternativa para correr modelos locales y comparar enfoques.

Principio de seguridad del diseño

La IA no debe tener control absoluto sobre el PLC. Por eso se estableció un contrato de salida: el modelo solo puede devolver una configuración lógica limitada a entradas, salidas, timers, contadores y modos permitidos. Después, el sistema valida antes de renderizar o aplicar. 

Texto del usuario:
"Con I1 prende la lámpara verde y con I2 apágala"

Salida esperada de la IA:
{
  "name": "Enclavamiento verde",
  "device_profile": "maletin_basico",
  "reset_before": true,
  "system": { "enable": true, "global_stop": null },
  "outputs": [
    {
      "output": "Q10",
      "logic": { "mode": "enclavado", "start": "I1", "stop": "I2" },
      "timer": null,
      "counter": null,
      "expr": "(I1 + Q10) * !I2",
      "comment": "I1 enciende y enclava la verde; I2 la apaga"
    }
  ]
}
10 · Créditos y herramientas de apoyo

Herramientas que apoyaron el desarrollo

Este proyecto fue desarrollado como parte de un trabajo académico de prospectiva tecnológica. Su avance fue posible gracias a la combinación de conocimientos de automatización, desarrollo web, programación en Python, pruebas con PLC y apoyo de herramientas de inteligencia artificial.

Claude Code

Herramienta clave en el desarrollo. Fue especialmente importante para estructurar código, refactorizar partes del proyecto, ordenar módulos y hacer posible la integración completa entre frontend, backend y flujo de generación.

ChatGPT

Usado como apoyo para definir arquitectura, explicar conceptos de PLC, generar y revisar fragmentos de código, proponer flujos de trabajo, documentar decisiones y ordenar el desarrollo paso a paso.

Gemini

Utilizado para resolver algunas dudas puntuales y contrastar ideas durante el proceso de diseño y validación del proyecto.

Groq

API utilizada para transcripción de voz con Whisper y para generación de respuestas estructuradas mediante LLM, especialmente en el flujo de texto a JSON lógico.

Ollama

Herramienta considerada y probada como alternativa para ejecutar modelos locales y explorar asistencia técnica sin depender completamente de servicios externos.

Agradecimiento especial a Claude Code: gran parte de la consolidación técnica del proyecto, la organización del código y la integración práctica fueron posibles gracias a su apoyo durante el desarrollo.
